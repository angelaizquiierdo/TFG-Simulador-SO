description: Guía para implementar y registrar un algoritmo de planificación personalizado.
---

## Interfaz `IAlgorithm`

Todo algoritmo debe implementar la interfaz `IAlgorithm` exportada por el módulo:

```ts
import type { IAlgorithm, ReadyProcess } from 'cpu-scheduler';

class MiAlgoritmo implements IAlgorithm {
  readonly name = 'mi-algoritmo';
  readonly preemptionMode = 'none'; // 'none' | 'on-better' | 'on-quantum'
  readonly requires = { priority: false, quantum: false };

  select(ready: readonly ReadyProcess[]): ReadyProcess {
    if (ready.length === 0) throw new Error('Cola vacía');
    // Lógica de selección: devuelve el proceso elegido
    return ready[0]!;
  }
}
```

### Valores de `preemptionMode`

| Valor | Comportamiento |
|-------|----------------|
| `'none'` | No expropiativo: el motor solo llama a `select()` cuando la CPU queda libre. |
| `'on-better'` | Expropiativo: el motor reevalúa en cada tick; si `select()` devuelve un proceso distinto al actual, expropia. |
| `'on-quantum'` | Round Robin: expropia al agotar el quantum y reencola el proceso activo. |

## Registrar el algoritmo

Llama a `register()` antes de montar el componente:

```ts
import { register } from 'cpu-scheduler';
import { MiAlgoritmo } from './mi-algoritmo';

register(new MiAlgoritmo());
```

A partir de ese momento puedes usarlo por nombre:

```tsx
import { SimulationProvider } from 'cpu-scheduler';

<SimulationProvider
  algorithm="mi-algoritmo"
  processes={[...]}
/>
```

## Reglas del método `select()`

- Recibe la cola de listos (`ready`) ya ordenada por el motor (criterio principal → menor `arrival_time` → menor `id`).
- Debe devolver uno de los elementos de `ready` (no puede crear procesos nuevos).
- Si `ready` está vacía, debe lanzar un error (el motor nunca llama a `select()` con cola vacía, pero es buena práctica protegerse).
- No debe acceder a estado externo, `Math.random()` ni `Date.now()`: el simulador debe ser determinista.
