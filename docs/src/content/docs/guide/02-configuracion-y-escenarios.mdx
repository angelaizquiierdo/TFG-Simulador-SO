## Definición de Procesos (`Process`)

Cada proceso tiene los siguientes campos:

| Campo | Tipo | Obligatorio | Descripción |
|---|---|---|---|
| `id` | `string` | Sí | Identificador único del proceso (p. ej. `"P1"`) |
| `arrival_time` | `number` (entero ≥ 0) | Sí | Tick en que el proceso llega a la cola de listos |
| `burst_time` | `number` (entero > 0) | Sí | Tiempo total de CPU que necesita el proceso |
| `priority` | `number` (entero > 0) | No | Prioridad. Menor número = mayor prioridad. Solo para algoritmos de prioridad. |
| `io` | `IOOperation[]` | No | Operaciones de E/S. Solo en Round Robin Virtual. |

```ts
const processes: Process[] = [
  { id: 'P1', arrival_time: 0, burst_time: 5 },
  { id: 'P2', arrival_time: 2, burst_time: 3, priority: 1 },
];
```

## Modelo de Entrada/Salida (E/S)

**Solo el algoritmo Round Robin Virtual modela E/S** (`requires.io = true`). Los demás
algoritmos ignoran el campo `io` aunque esté declarado.

Cada operación de E/S en el array `io` tiene:

| Campo | Tipo | Descripción |
|---|---|---|
| `io_entry` | `number` (entero > 0) | CPU acumulada que el proceso ejecuta antes de bloquearse. Debe ser estrictamente menor que `burst_time`. |
| `io_time` | `number` (entero > 0) | Duración del servicio de E/S en ticks. |
| `device` | `string` (opcional) | Identificador del dispositivo. Por defecto `"0"`. |

**Restricciones:**
- Los valores de `io_entry` deben ser **estrictamente crecientes** dentro del array.
- `io_entry > 0` y `io_entry < burst_time`.
- `io_time > 0`.

```ts
// Proceso con dos operaciones de E/S
{ id: 'A', arrival_time: 0, burst_time: 9, io: [
  { io_entry: 1, io_time: 2 },   // se bloquea al ejecutar 1 tick de CPU
  { io_entry: 5, io_time: 3 },   // segunda E/S al ejecutar 5 ticks acumulados
]}
```

## Configuración del Algoritmo (`RunConfig`)

El algoritmo se indica por su identificador de registro:

| Identificador | Algoritmo |
|---|---|
| `"fcfs"` | First Come First Served |
| `"sjf"` | Shortest Job First |
| `"ljf"` | Longest Job First |
| `"priority-np"` | Prioridad (no expropiativa) |
| `"srtf"` | Shortest Remaining Time First |
| `"round-robin"` | Round Robin |
| `"priority-p"` | Prioridad (expropiativa) |
| `"vrr"` | Round Robin Virtual |
| `"mlfq"` | Cola de Realimentación Multinivel |

### Parámetros dinámicos

| Parámetro | Tipo | Algoritmos | Descripción |
|---|---|---|---|
| `quantum` | `number` (entero > 0) | `round-robin`, `vrr` | Tiempo máximo de CPU antes de expropiar |
| `quanta` | `number[]` | `mlfq` | Array de quantums por nivel (uno por nivel) |
| `boostInterval` | `number` (entero > 0, opcional) | `mlfq` | Cada cuántos ticks se produce un priority boost. Sin valor = sin boost. |

```tsx
// Round Robin con quantum 2
<SimulationProvider algorithm="round-robin" processes={processes} params={{ quantum: 2 }}>

// MLFQ con 2 niveles y boost cada 8 ticks
<SimulationProvider algorithm="mlfq" processes={processes} params={{ quanta: [2, 4], boostInterval: 8 }}>
```

## Manipulación de Escenarios

### Edición Estándar

Los componentes `ProcessForm` y `AlgorithmParamsForm` permiten modificar la configuración
desde la interfaz:

- **`ProcessForm`**: panel desplegable, cerrado por defecto. Al editar cualquier campo, la
  simulación se **rederiva al instante** (sin botón "Aplicar").
- **`AlgorithmParamsForm`**: mantiene un estado `draft` (lo que el usuario escribe) y un
  estado `applied` (lo que se usa en la simulación). La rederivación ocurre solo al pulsar
  **"Aplicar"**, no tecla a tecla. Si el algoritmo no tiene parámetros configurables, el
  componente no renderiza nada.

### Análisis What-If

El simulador soporta rederivar desde un tick intermedio con una configuración distinta:

1. El usuario pausa la reproducción en el tick `T`.
2. Modifica un parámetro (p. ej. cambia el quantum o añade un proceso).
3. El motor llama internamente a `runFrom(state, config, allProcesses)` con el estado en
   el tick `T`, generando una **línea temporal alternativa** a partir de ese punto.

El historial original no se destruye: el what-if genera un nuevo historial desde `T` en
adelante.

### Inyección en Vivo

Se puede añadir un proceso nuevo mientras la simulación está en marcha, siempre que:

- `arrival_time >= tick_actual` (no se puede insertar un proceso con llegada en el pasado).

Si se viola esta condición, el motor lanza un error descriptivo.

## Persistencia

El escenario activo (procesos, algoritmo, parámetros) se guarda automáticamente en
`sessionStorage` del navegador con la clave `scheduler-scenario:<nombre-algoritmo>`.

- Los datos **se conservan al recargar la página**.
- Los datos **se descartan al cerrar la pestaña** (no es `localStorage`).
- Cada página de algoritmo tiene su propia clave: navegar entre páginas no mezcla
  escenarios.
- Existe una acción de **reset** que restaura el escenario al estado inicial definido por
  las props.
